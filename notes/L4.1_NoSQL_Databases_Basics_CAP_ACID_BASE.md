## 1. Relational Databases

A **relational database** is a system for persistent storage of digital data that provides a higher level of abstraction than raw files.

### 1.1 Core Properties

- Supports **CRUD** operations: **C**reate, **R**ead/Query, **U**pdate, **D**elete
- Provides both **programmatic** and **UI-based** access
- Uses a **formal query language** (SQL) for data manipulation
- Managed by a **Database Management System (DBMS)**

### 1.2 Data Model

Data in a relational database is organized as **tables** (also called **relations**):

```
┌──────────────────────────────────────────────────────────────────────┐
│                     RELATIONAL DATA MODEL                            │
│                                                                      │
│   Table = Relation                                                   │
│   ┌─────────┬──────────┬──────────┬──────────┬──────────┐           │
│   │   SNO   │  SNAME   │  STATUS  │   CITY   │  WEIGHT  │  ← Columns│
│   │  (key)  │  (text)  │  (int)   │  (text)  │ (decimal)│   (Fields)│
│   ├─────────┼──────────┼──────────┼──────────┼──────────┤           │
│   │   S1    │  Smith   │    20    │  London  │   12.0   │  ← Row    │
│   │   S2    │  Jones   │    10    │  Paris   │    8.5   │  (Record/ │
│   │   S3    │  Blake   │    30    │  Paris   │   15.2   │   Tuple)  │
│   │   S4    │  Clark   │    20    │  London  │   10.0   │           │
│   └─────────┴──────────┴──────────┴──────────┴──────────┘           │
│                                                                      │
│   Key Properties:                                                    │
│   • Columns have a DATA TYPE (decimal, text, int, etc.)             │
│   • Columns can have INTEGRITY CONSTRAINTS (e.g., uniqueness)       │
│   • SNO is a UNIQUE KEY — no two rows share the same SNO            │
│   • ALL rows have ALL columns present (strict schema)               │
└──────────────────────────────────────────────────────────────────────┘
```

### 1.3 Linking Tables (Foreign Keys)

Tables can be **linked** using key constraints:

```
┌───────────────────┐          ┌───────────────────────────┐
│    SUPPLIERS       │          │      SHIPMENTS             │
│                    │          │                             │
│  SNO (PK) │ SNAME  │    ┌──→ │  SNO (FK) │ PNO │ QTY     │
│───────────┼────────│    │    │───────────┼─────┼─────────│
│  S1       │ Smith  │────┘    │  S1       │ P1  │  300    │
│  S2       │ Jones  │────┐    │  S1       │ P2  │  200    │
│  S3       │ Blake  │    └──→ │  S2       │ P1  │  400    │
└───────────────────┘          └───────────────────────────┘

   PK = Primary Key                FK = Foreign Key
```

- **Foreign keys** in one table point to **primary keys** in another
- This **avoids data duplication** and **ensures consistency** — if supplier S1's name changes, it only changes in one place
- The schema is defined using **Data Definition Language (DDL)**

**Example:** Instead of repeating "Smith" in every shipment record, we store it once in SUPPLIERS and reference it via `SNO`.

## 2. DBMS — Logical View vs. Physical Layout

A **Database Management System (DBMS)** manages the database and separates the logical view from the physical layout to achieve **data independence** for applications.

### 2.1 Logical Layer vs. Physical Layer

```
┌──────────────────────────────────────────────────────────────────────┐
│                    DBMS: TWO-LAYER ABSTRACTION                       │
│                                                                      │
│   ┌────────────────────────────────────────────────────────────┐     │
│   │                    LOGICAL LAYER                            │     │
│   │                                                             │     │
│   │  • Tables, Columns, Rows                                   │     │
│   │  • Declarative queries (SQL)                                │     │
│   │  • "WHAT data do I want?"                                   │     │
│   │  • Application developers work at this layer                │     │
│   └──────────────────────────┬─────────────────────────────────┘     │
│                              │ Data Independence                     │
│   ┌──────────────────────────▼─────────────────────────────────┐     │
│   │                    PHYSICAL LAYER                            │     │
│   │                                                             │     │
│   │  • Blocks, Files, B-tree Indexes                            │     │
│   │  • Query Planners, Optimizers, Execution Engines            │     │
│   │  • "HOW is the data stored and accessed?"                   │     │
│   │  • DBA and DBMS internals manage this layer                 │     │
│   └────────────────────────────────────────────────────────────┘     │
│                                                                      │
│   KEY INSIGHT: You can change the physical layout (add indexes,      │
│   move to SSD, partition data) WITHOUT changing the application      │
│   code. This is "data independence."                                 │
└──────────────────────────────────────────────────────────────────────┘
```

## 3. ACID Properties

**ACID** is the correctness model for traditional relational databases. It defines four guarantees that every transaction must satisfy.

### 3.1 The Four Properties

| Property | Description | Example |
|---|---|---|
| **Atomicity** | Even if a transaction has multiple operations, it either completes fully (**commit**) or has no effect at all (**abort/rollback**). | Transferring $100 from Account A to B: either both the debit from A and credit to B happen, or neither does. |
| **Consistency** | A transaction that runs on a correct database leaves it in a correct ("consistent") state — all integrity constraints hold. | If a rule says "balance ≥ 0", no committed transaction can violate it. |
| **Isolation** | Each transaction appears to run **all by itself** — concurrency is hidden from the application. | If two users update the same row simultaneously, the result is as if one ran before the other (serializable execution). |
| **Durability** | Once a transaction commits, its updates **cannot be lost** or rolled back, even if the system crashes. | After a bank transfer commits, a power failure won't undo it. |

### 3.2 Why ACID Eases Development

```
┌──────────────────────────────────────────────────────────────────────┐
│              WHY ACID MAKES LIFE EASIER                              │
│                                                                      │
│  WITHOUT ACID (developer must handle):                               │
│  ┌─────────────────────────────────────────────────┐                 │
│  │  • Partial state: "Tony is retired, but some    │                 │
│  │    accounts still show him as account rep"       │                 │
│  │  • Seeing a concurrent transaction's half-done   │                 │
│  │    state → possible crashes, corruption          │                 │
│  │  • Manual retry logic, manual undo on failure    │                 │
│  └─────────────────────────────────────────────────┘                 │
│                                                                      │
│  WITH ACID (database guarantees):                                    │
│  ┌─────────────────────────────────────────────────┐                 │
│  │  ✓ No partial state: all-or-nothing execution    │                 │
│  │  ✓ No transient inconsistencies visible          │                 │
│  │  ✓ Serial & serializable execution: concurrency  │                 │
│  │    with hidden side-effects                      │                 │
│  │  ✓ Committed data survives crashes               │                 │
│  └─────────────────────────────────────────────────┘                 │
└──────────────────────────────────────────────────────────────────────┘
```

**Key takeaway**: ACID offers **concurrency** while **hiding side-effects** through serial & serializable execution.

## 4. Limitations of RDBMS

Despite its strengths, the relational model has significant limitations when dealing with modern data and scale:

### 4.1 Not Well-Suited for Unstructured/Semi-Structured Data

| Problem | Why RDBMS Struggles |
|---|---|
| Rows with different number of columns | Relational tables require a fixed schema — every row must have all columns |
| Schemas that change often | `ALTER TABLE` operations can be expensive and require downtime |
| Columns with unknown types | Strict typing prevents storing arbitrary data |
| Free text data | SQL full-text search is limited compared to specialized search engines |
| Non-text data (image, audio, video) | BLOBs are supported but not efficiently queried |
| Semi-structured data (JSON, XML) | Doesn't map well to rigid table structures |
| Social networks, graphs | Joins across many tables become extremely expensive |

### 4.2 ACID Limits Scalability

```
┌──────────────────────────────────────────────────────────────────────┐
│                ACID vs. SCALABILITY TRADEOFF                         │
│                                                                      │
│    RDBMS can SCALE UP (bigger single machine)                        │
│    but NOT easily SCALE OUT (many machines)                          │
│                                                                      │
│    WHY?                                                              │
│    ┌──────────────────────────────────────────────┐                  │
│    │ • Strong consistency requires coordination   │                  │
│    │   across all nodes (2-phase commit, locking) │                  │
│    │ • More nodes = more messages = more latency  │                  │
│    │ • Deadlocks increase non-linearly with nodes │                  │
│    │ • Scale-up has a ceiling (hardware limits)   │                  │
│    └──────────────────────────────────────────────┘                  │
│                                                                      │
│    Scale-UP:  1 machine → bigger machine   (limited, expensive)     │
│    Scale-OUT: 1 machine → N machines       (desired, but hard       │
│                                              with ACID)             │
└──────────────────────────────────────────────────────────────────────┘
```

## 5. The Evolution of Cloud Architecture

### 5.1 How Did Today's Cloud Evolve?

Before ~2005, data centers were designed for high availability using traditional approaches. Amazon had especially large data centers to serve web requests (before the AWS cloud model). But as load grew, their system became unreliable — the core problem was **scaling**.

### 5.2 The Yahoo Experiment — Speed = Revenue

Yahoo ran an alpha/beta experiment and discovered a critical insight:

```
┌──────────────────────────────────────────────────────────────────────┐
│              THE YAHOO SPEED EXPERIMENT                              │
│                                                                      │
│   Response Time          User Behavior                               │
│   ─────────────          ──────────────                              │
│   < 100ms                Users are HAPPY, purchase rate is high      │
│   100ms → 200ms          Purchase rates NOTICEABLY DROP              │
│   200ms → 300ms          Further decline                             │
│   300ms+                 Significant revenue loss                    │
│                                                                      │
│   CONCLUSION: Speed at scale determines revenue.                    │
│   Revenue shapes technology → arms race to speed up the cloud.      │
└──────────────────────────────────────────────────────────────────────┘
```

Every **100ms of delay** caused a noticeable drop in purchases. This insight drove Amazon (starting ~2006) to completely reinvent data center computing.

### The "Obvious" Solution (That Doesn't Work)

## 10. Why a Single Big Service Fails at Scale

### 10.1 The Problem

Until ~2005, "one server" (or a small cluster of 2-16 machines) could keep up for services like Amazon's shopping cart. But as the cloud grew, this form of scaling broke:

> Companies threw unlimited money at the issue but critical services like databases still became hopelessly overloaded, crashed, or fell far behind.

### 10.2 Jim Gray's Famous Paper (1996)

**Jim Gray** (Jan 1944 – Jan 2007) and colleagues at Microsoft anticipated this problem as early as 1996 in their landmark paper:

> *"The dangers of replication and a solution."*  
> Jim Gray, Pat Helland, Patrick O'Neil, Dennis Shasha, ACM SIGMOD 1996.

**Basic message: Divide and conquer is really the only option.**

The paper uses a "chalkboard analysis" to reason about scaling for any system that behaves like a database — or any replicated storage layer where you want strong guarantees of data consistency.

---

## 11. The Core Issue — Lock-Based Consistency at Scale

### 11.1 Setup

The paper assumes:
- The goal is some form of **lock-based consistency** (modeled as database serializability)
- Applications use **read locks** and **write locks**
- We want to accommodate more load by **adding more servers** (weak scaling)

```
┌──────────────────────────────────────────────────────────────────────┐
│                     BASIC SETUP                                      │
│                                                                      │
│   ┌────────┐ ┌────────┐ ┌────────┐                                  │
│   │ App 1  │ │ App 2  │ │ App 3  │   ← T concurrent requests        │
│   └───┬────┘ └───┬────┘ └───┬────┘     (T = 3 here, but could be    │
│       │          │          │            much larger)                 │
│       ▼          ▼          ▼                                        │
│   ┌──────────────────────────────────┐                               │
│   │         THE DATABASE             │  ← Single instance            │
│   └──────────────────────────────────┘                               │
│                                                                      │
│   For scalability, replicate across N servers:                        │
│                                                                      │
│   ┌────────┐ ┌────────┐ ┌────────┐                                  │
│   │ App 1  │ │ App 2  │ │ App 3  │   ← T concurrent requests        │
│   └───┬────┘ └───┬────┘ └───┬────┘                                  │
│       │          │          │                                        │
│       ▼          ▼          ▼                                        │
│   ┌────────┐ ┌────────┐ ┌────────┐                                  │
│   │Server 1│ │Server 2│ │Server N│   ← N replicated servers          │
│   └────────┘ └────────┘ └────────┘                                  │
│                                                                      │
│   More replicas → MORE need for read and write locks                 │
└──────────────────────────────────────────────────────────────────────┘
```

### 11.2 What Should Be the Goals?

A scalable system needs to handle **more T's** (transactions) by adding to **N** (servers) — this is **Weak Scaling**.

**What actually happens:**
- The work the servers must do increases **non-linearly** with T
- Deadlocks occur as N increases, causing **feedback** (reissued transactions get done more than once)

**Example from the paper:**
> If 3 servers (N=3) could do 1,000 TPS (transactions per second), with 5 servers the rate might drop to **300 TPS** — purely because of deadlocks forcing abort/retry.

This is the opposite of what we want! More servers → _worse_ performance.

### 11.3 Why Do Services Slow Down at Scale?

The paper identified two main issues:

```
┌──────────────────────────────────────────────────────────────────────┐
│           WHY REPLICATED SYSTEMS SLOW DOWN AT SCALE                  │
│                                                                      │
│   ISSUE 1: LOCK CONTENTION                                           │
│   ┌──────────────────────────────────────────────────────┐           │
│   │  More concurrent tasks  →  higher probability that   │           │
│   │  multiple tasks try to access the SAME object  →     │           │
│   │  they must WAIT for locks  →  throughput drops        │           │
│   └──────────────────────────────────────────────────────┘           │
│                                                                      │
│   ISSUE 2: ABORT & RETRY                                             │
│   ┌──────────────────────────────────────────────────────┐           │
│   │  Many consistency mechanisms have optimistic          │           │
│   │  behavior  →  occasionally must BACK OUT and RETRY    │           │
│   │  Deadlocks also cause abort/retry sequences  →        │           │
│   │  wasted work compounds  →  system collapses           │           │
│   └──────────────────────────────────────────────────────┘           │
│                                                                      │
│   KEY FINDING: Deadlock/reconciliation rate grows CUBICALLY          │
│   with the number of nodes!                                          │
│                                                                      │
│   If N increases 10x  →  deadlocks increase ~1000x                  │
└──────────────────────────────────────────────────────────────────────┘
```

### 11.4 The Replication Taxonomy (from Jim Gray's Paper)

The paper identifies two dimensions of replication:

| | **Eager** (synchronous) | **Lazy** (asynchronous) |
|---|---|---|
| **Group** (update anywhere) | All replicas updated atomically in one transaction. Serializable. But deadlocks explode with scale. | One replica updated; others catch up later. Risk of stale reads, reconciliation failures, **system delusion**. |
| **Master** (primary copy) | Master processes all writes; replicas updated as part of same transaction. Reduces conflicts. | Master updated first; replicas updated asynchronously. Much more stable, but replicas may have stale data. |

**Key insight:** *Update-anywhere-anytime-anyway replication is unstable.* The only stable replication strategy at scale is **master-copy** with controlled propagation.

**The checking account analogy from the paper:**

> Consider a joint checking account shared with your spouse, with $1,000 in it. It's replicated in three places: your checkbook, your spouse's checkbook, and the bank's ledger.
>
> - **Eager replication** prevents you from overdrawing — the bank, you, and your spouse always agree.
> - **Lazy replication** allows both of you to write checks totaling $1,000 each (total $2,000 withdrawn!). When these checks arrive at the bank, someone must **reconcile** the inconsistency.
>
> With master replication, only the bank's updates count — reducing these problems.

## 12. Sharding — "That One Good Option"

### 12.1 The Conclusion

Back in 1996, Jim Gray concluded:

> **You are forced to shard the database into a large set of much smaller databases, with distinct data in each.**

By 2007, Gray had demonstrated that for every problem he encountered, it was possible to devise a sharded solution where transactions only touched a **single shard** at a time.

### 12.2 How Sharding Works

```
┌──────────────────────────────────────────────────────────────────────┐
│                        SHARDING                                      │
│                                                                      │
│   A transaction: read/write to some (key, value) tuple               │
│   Shard assignment: Hash(key) % K                                    │
│                                                                      │
│   ┌────────┐                                                         │
│   │  App   │─── read(key) ──→ Hash(key) % K = 1 ──→ Shard 1        │
│   └────────┘                                                         │
│                                                                      │
│   K shards distributed across servers:                               │
│                                                                      │
│   Server 1          Server 2          Server 3                       │
│   ┌─────────┐       ┌─────────┐       ┌─────────┐                   │
│   │ Shard 0 │       │ Shard 2 │       │ Shard 4 │                   │
│   │ Shard 1 │       │ Shard 3 │       │ Shard 5 │                   │
│   └─────────┘       └─────────┘       └─────────┘                   │
│                                                                      │
│   3 servers × 2 shards each = 6 shards total                        │
└──────────────────────────────────────────────────────────────────────┘
```

### 12.3 Why Single-Shard Transactions Scale

If a transaction does all its work at **just one shard**, never needing to access two or more, then:
- **No locks** between shards are required
- **No 2-phase commit** needed
- This scales **very well**

```
┌──────────────────────────────────────────────────────────────────────┐
│              SINGLE-SHARD TRANSACTIONS: SCALABLE ✓                   │
│                                                                      │
│   Each transaction accesses just ONE shard:                          │
│                                                                      │
│   Txn A ──→ Shard 1     (no cross-shard coordination)               │
│   Txn B ──→ Shard 3     (independent, parallel)                     │
│   Txn C ──→ Shard 0     (no locks between shards)                   │
│                                                                      │
│   Result: Linear scalability! Add more shards → more throughput     │
└──────────────────────────────────────────────────────────────────────┘
```

### 12.4 Why Multi-Shard Transactions Fail

Transactions that touch **multiple shards** need locks and 2-phase commit. Jim Gray's analysis applies: as scale increases, performance collapses.

```
┌──────────────────────────────────────────────────────────────────────┐
│            MULTI-SHARD TRANSACTIONS: COLLAPSE ✗                      │
│                                                                      │
│   Txn X needs to read Shard 1 AND write Shard 3                     │
│     → Needs distributed lock across both shards                     │
│     → Needs 2-phase commit across both servers                      │
│     → As we add more shards: deadlocks, contention explode          │
│                                                                      │
│   CONCLUSION: Design your data model so transactions                 │
│   only need ONE shard!                                               │
└──────────────────────────────────────────────────────────────────────┘
```

### 12.5 Sharding with Replication

For reliability, we also **replicate** each shard:

```
┌──────────────────────────────────────────────────────────────────────┐
│              SHARDING + REPLICATION                                   │
│                                                                      │
│   K shards, each with 2 replicas:                                    │
│                                                                      │
│   Server 1          Server 2          Server 3                       │
│   ┌─────────┐       ┌─────────┐       ┌─────────┐                   │
│   │Shard 0  │       │Shard 0  │       │Shard 1  │                   │
│   │(primary)│       │(replica)│       │(primary)│                   │
│   │Shard 1  │       │Shard 2  │       │Shard 2  │                   │
│   │(replica)│       │(primary)│       │(replica)│                   │
│   └─────────┘       └─────────┘       └─────────┘                   │
│                                                                      │
│   PROBLEM: Write operations must update BOTH the primary             │
│   and replica of a shard → causes lock contention again!             │
│                                                                      │
│   Write to Shard 0 → must update primary (Server 1)                  │
│                       AND replica (Server 2)                         │
│                     → Lock contention between replicas               │
└──────────────────────────────────────────────────────────────────────┘
```

**Data sharding helps scale the database, but strong consistency can still cause a bottleneck and limit availability (à la ACID).**

## 13. Questions About Consistency

After sharding with replication, a key question emerges:

> **What about weaker consistency?**

The idea is:
- Tasks have some kind of **read-only data** computed "last night"
- Different replicas can **temporarily** have different contents
- Updates can be **enqueued** for offline processing
- Tasks might **guess** at the effect of updates, but the offline version will "win" if a conflict occurs

This leads to the **BASE** methodology...

## 14. BASE Methodology

### 14.1 Origin and Philosophy

**BASE** was invented at **eBay**, adopted by **Amazon** and others. It stands for:

| Letter | Meaning |
|---|---|
| **BA** | **B**asically **A**vailable |
| **S** | **S**oft State |
| **E** | **E**ventual Consistency |

> **BASE = "CAP in practice"**  
> Philosophy: *"Use CAP. You can clean up later."*

How it works: cache data but **don't worry about cache entries getting stale**.

### 14.2 Basically Available

```
┌──────────────────────────────────────────────────────────────────────┐
│                    BASICALLY AVAILABLE                                │
│                                                                      │
│   GOAL: Promote RAPID responses                                      │
│                                                                      │
│   • Partitioning faults are RARE in data centers                     │
│   • Crashes force isolated machines to reboot                        │
│   • Need rapid responses even when some replicas on the              │
│     critical path can't be contacted                                 │
│                                                                      │
│   Example:                                                           │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐                       │
│   │ Replica 1│    │ Replica 2│    │ Replica 3│                       │
│   │  (fast)  │    │  (SLOW)  │    │ (CRASHED)│                       │
│   └────┬─────┘    └──────────┘    └──────────┘                       │
│        │                                                             │
│        ▼                                                             │
│   Return response from Replica 1 immediately!                        │
│   Don't wait for all replicas.                                       │
└──────────────────────────────────────────────────────────────────────┘
```

### 14.3 Soft State

```
┌──────────────────────────────────────────────────────────────────────┐
│                      SOFT STATE                                      │
│                                                                      │
│   • Runs in FIRST TIER (lightweight web-facing servers)              │
│   • CANNOT store permanent data                                     │
│   • Restarts in a "CLEAN" state after a crash                        │
│                                                                      │
│   To remember data, two options:                                     │
│   ┌──────────────────────────────────────────────────────┐           │
│   │ Option 1: Replicate in memory in enough copies to    │           │
│   │           never lose ALL copies in any crash          │           │
│   │                                                      │           │
│   │ Option 2: Pass data to another service that keeps    │           │
│   │           "hard state" (durable storage in Tier 2)   │           │
│   └──────────────────────────────────────────────────────┘           │
│                                                                      │
│   If a Tier 1 node crashes → it just restarts clean,                │
│   fetches state from Tier 2 or peers                                │
└──────────────────────────────────────────────────────────────────────┘
```

### 14.4 Eventual Consistency

```
┌──────────────────────────────────────────────────────────────────────┐
│                   EVENTUAL CONSISTENCY                                │
│                                                                      │
│   OK to send "OPTIMISTIC" answers to external client:                │
│                                                                      │
│   ✓ Send reply to user BEFORE finishing the operation                │
│   ✓ Use cached data (WITHOUT staleness check)                        │
│   ✓ GUESS the outcome of an update                                   │
│   ✓ SKIP LOCKS, hoping no conflicts happen                           │
│   ✓ Later, if needed, correct inconsistencies in                     │
│     an OFFLINE CLEANUP activity                                      │
│                                                                      │
│   Timeline:                                                          │
│   ┌────────────────────────────────────────────────────┐             │
│   │ t=0: Client writes value V=5 to Replica 1         │             │
│   │ t=0: Reply "success" to client immediately        │             │
│   │ t=1: Replica 2 still has old value V=3             │             │
│   │ t=2: Replica 3 still has old value V=3             │             │
│   │ t=5: Asynchronous propagation → Replica 2 gets V=5│             │
│   │ t=8: Asynchronous propagation → Replica 3 gets V=5│             │
│   │ t=8: ALL replicas EVENTUALLY consistent (V=5)      │             │
│   └────────────────────────────────────────────────────┘             │
│                                                                      │
│   ⚠ WARNING: Developer ends up thinking hard and working hard!      │
│   The system pushes complexity from the DB to the APPLICATION.       │
└──────────────────────────────────────────────────────────────────────┘
```

**Example:** When you add an item to your Amazon shopping cart, the response is instant. Behind the scenes, the update may not have reached all replicas yet. If you check from a different device immediately, you _might_ not see the item — but within seconds, all replicas will converge.

## 15. CAP Theorem

### 15.1 Origin

At Berkeley, **Eric Brewer** captured the consistency-availability tradeoff with a "theorem":

> CAP stands for **Consistency**, **Availability**, and **Partition Tolerance**.

The theoretically "best" solution often brings heavy costs. Consistency requires coordination, coordination takes time and involves node-to-node dialogue. But to earn the most money, you need the **fastest possible responses**. Brewer concluded: **you might have to relax consistency**.

### 15.2 The Three Properties

```
┌──────────────────────────────────────────────────────────────────────┐
│                      CAP THEOREM                                     │
│                                                                      │
│   "You can have only 2 of 3 properties"                              │
│                                                                      │
│              Consistency (C)                                         │
│                  /\                                                   │
│                 /  \                                                  │
│                /    \                                                 │
│         CA ──/──────\── CP                                           │
│             /   YOU   \                                               │
│            /   CAN'T   \                                             │
│           /   HAVE ALL  \                                            │
│          /     THREE     \                                           │
│         /________________\                                           │
│   Availability (A) ──── Partition Tolerance (P)                      │
│              AP                                                      │
│                                                                      │
│   ┌─────────────────────────────────────────────────────────┐        │
│   │ C: The system responds using the MOST CURRENT values.   │        │
│   │    Conflicting updates are performed in some specific   │        │
│   │    order by ALL replicas. Queries see a "single system" │        │
│   │    and are up to date.                                  │        │
│   │                                                         │        │
│   │ A: The system is RAPIDLY responsive and will self-      │        │
│   │    repair if some single component fails, ASAP.         │        │
│   │    If too many components fail, availability is lost.   │        │
│   │                                                         │        │
│   │ P: The cloud should continue to respond even if its     │        │
│   │    services are temporarily unable to talk to each      │        │
│   │    other (network issues, entire services down).        │        │
│   └─────────────────────────────────────────────────────────┘        │
│                                                                      │
│   ACID systems choose: C + P (sacrifice Availability)                │
└──────────────────────────────────────────────────────────────────────┘
```

### 15.3 The Three CAP Combinations

| Combination | What You Get | What You Lose | Examples |
|---|---|---|---|
| **CA** (Consistency + Availability) | Single-copy consistency with fast responses | Cannot handle network partitions | Traditional RDBMS (single-site) |
| **CP** (Consistency + Partition Tolerance) | Data is always correct, survives partitions | May be unavailable during partitions (blocks requests) | MongoDB, HBase, Spanner |
| **AP** (Availability + Partition Tolerance) | Always responds, survives partitions | Responses may be stale or inconsistent | Dynamo, Cassandra, CouchDB |

### 15.4 Why "2 of 3" Is Misleading (Brewer's 2012 Revisit)

Twelve years after the original theorem, Brewer clarified important nuances:

```
┌──────────────────────────────────────────────────────────────────────┐
│              WHY "2 OF 3" IS MISLEADING                              │
│                                                                      │
│   1. Partitions are RARE                                             │
│      → No reason to forfeit C or A when system is NOT partitioned   │
│      → Most of the time, you CAN have all three!                    │
│                                                                      │
│   2. The C/A choice is NOT system-wide                               │
│      → Different SUBSYSTEMS can make different choices               │
│      → The choice can change per OPERATION or per DATA item          │
│      → Shopping cart (AP) vs. payment processing (CP)               │
│                                                                      │
│   3. All three properties are CONTINUOUS, not binary                 │
│      → Availability: 0% to 100%                                    │
│      → Consistency: many levels (linearizable → eventual)           │
│      → Partitions: nuances, disagreement about existence            │
│                                                                      │
│   MODERN CAP GOAL: Maximize combinations of C and A that make       │
│   sense for the specific application. Plan for partitions.           │
└──────────────────────────────────────────────────────────────────────┘
```

### 15.5 CAP-Latency Connection

In practice, **latency and partitions are deeply related**:

> Operationally, the essence of CAP takes place during a **timeout** — a period when the program must make a **partition decision**:
> 1. **Cancel** the operation → decrease availability, or
> 2. **Proceed** with the operation → risk inconsistency

Retrying communication (e.g., via Paxos or 2-phase commit) just delays the decision. At some point, the system must choose. Retrying indefinitely means choosing C over A.

**Practical implication:** Systems with tighter latency bounds will enter "partition mode" more often, even when the network is merely slow (not actually partitioned).

### 15.6 Managing Partitions (Three-Step Strategy)

Brewer proposes an explicit strategy for dealing with partitions:

```
┌──────────────────────────────────────────────────────────────────────┐
│              THREE-STEP PARTITION MANAGEMENT                         │
│                                                                      │
│   Time ──────────────────────────────────────────────►               │
│                                                                      │
│   State: S   │  State: S1  /  State: S2   │  State: S'              │
│   (consistent)│  (side 1)  /  (side 2)    │  (reconciled)           │
│              │            /               │                          │
│   ───────────┤───────────/────────────────┤──────────                │
│              │  PARTITION MODE            │  RECOVERY                │
│              │                            │                          │
│   STEP 1: DETECT the partition                                       │
│   • System times out → enters partition mode                        │
│   • Both sides continue operating independently                     │
│                                                                      │
│   STEP 2: ENTER PARTITION MODE                                       │
│   • Limit or modify operations that might violate invariants         │
│   • Track history using VERSION VECTORS                              │
│   • Continue serving requests (maintain availability)                │
│   • Record operations for later reconciliation                       │
│                                                                      │
│   STEP 3: INITIATE RECOVERY when partition ends                      │
│   • Merge state S1 and S2 into consistent state S'                  │
│   • Compensate for mistakes made during partition                    │
│   • Use CRDTs / version vectors / commutative operations             │
│   • Fix externalized errors (e.g., email customer about duplicates) │
└──────────────────────────────────────────────────────────────────────┘
```

**Example — ATM Design:**

> An ATM must decide: prohibit withdrawals during a partition (safe but unavailable) or allow limited withdrawals?
>
> Modern ATMs use a **stand-in mode** (partition mode): they limit net withdrawals to $200. Below this limit, withdrawals work. Above it, they're denied.
>
> When the partition ends, the bank reconciles. If the balance went negative, the bank charges an overdraft fee — the **compensation** for the invariant violation during the partition.

## 16. ACID and CAP — Detailed Comparison

The C and A in ACID mean **different things** than in CAP. This is a common source of confusion.

### 16.1 Comparing Each ACID Property with CAP

| ACID Property | Relationship with CAP |
|---|---|
| **Atomicity (A)** | All systems benefit from atomic operations. When focusing on availability, **both sides** of a partition should still use atomic operations. Higher-level atomicity actually **simplifies recovery**. |
| **Consistency (C in ACID)** | In ACID, C means a transaction preserves all **database rules** (unique keys, referential integrity, etc.). In CAP, C means **single-copy consistency** — a strict **subset** of ACID consistency. ACID consistency **cannot be maintained across partitions**. Partition recovery must restore it. |
| **Isolation (I)** | Isolation is **at the core** of CAP. If the system requires ACID isolation, it can operate on **at most one side** during a partition. Serializability requires communication → fails across partitions. **Weaker definitions of correctness** are viable across partitions via compensation during recovery. |
| **Durability (D)** | No reason to forfeit durability. Although developers might avoid needing it via **soft state** (BASE). During partition recovery, it is possible to reverse durable operations that unknowingly violated an invariant. Given a durable history from both sides, errors can be detected and corrected. |

```
┌──────────────────────────────────────────────────────────────────────┐
│                  ACID vs. CAP at a Glance                            │
│                                                                      │
│   ACID:                           CAP:                               │
│   ┌───────────────────┐           ┌───────────────────┐              │
│   │ A = Atomicity     │           │ C = Single-copy   │              │
│   │ C = DB rule       │           │     consistency   │              │
│   │     preservation  │           │ A = High          │              │
│   │ I = Isolation     │           │     availability  │              │
│   │     (serializable)│           │ P = Partition     │              │
│   │ D = Durability    │           │     tolerance     │              │
│   └───────────────────┘           └───────────────────┘              │
│                                                                      │
│   ACID "C" ≠ CAP "C"                                                │
│   ACID C is broader (all DB rules)                                   │
│   CAP C is narrower (single-copy freshness only)                    │
│                                                                      │
│   KEY INSIGHT: Running ACID transactions on EACH SIDE of a          │
│   partition makes recovery EASIER — it enables a framework for      │
│   compensating transactions used for partition recovery.             │
└──────────────────────────────────────────────────────────────────────┘
```

### 16.2 Key Insight from Brewer (2012)

> "In general, running ACID transactions on each side of a partition makes recovery easier and enables a framework for compensating transactions that can be used for recovery from a partition."

In other words: you don't have to fully abandon ACID to use CAP. You can **run ACID locally** (within each partition) and reconcile across partitions during recovery.

## 18. Summary: Key Takeaways from Part 1

```
┌──────────────────────────────────────────────────────────────────────┐
│                 MODULE 4 PART 1: KEY TAKEAWAYS                       │
│                                                                      │
│  1. RDBMS provides ACID guarantees with structured data              │
│     but doesn't scale horizontally well.                             │
│                                                                      │
│  2. Modern cloud evolved from monolithic servers to                  │
│     MICROSERVICES with message buses and tiered architecture.        │
│                                                                      │
│  3. Jim Gray showed that replicated databases with strong            │
│     consistency COLLAPSE at scale (deadlocks grow cubically).        │
│                                                                      │
│  4. SHARDING (divide data into independent partitions) is            │
│     the only viable approach — but transactions must stay            │
│     within a single shard.                                           │
│                                                                      │
│  5. Adding replication for reliability reintroduces consistency      │
│     problems due to cross-replica coordination.                      │
│                                                                      │
│  6. CAP THEOREM: You can have at most 2 of {Consistency,            │
│     Availability, Partition Tolerance}, but in practice:             │
│     • Partitions are rare → you usually get all 3                   │
│     • The choice is PER-OPERATION, not system-wide                  │
│     • All three are continuous, not binary                           │
│                                                                      │
│  7. BASE (Basically Available, Soft State, Eventual Consistency)     │
│     is the practical alternative to ACID for distributed systems.   │
│     It trades immediate consistency for availability and speed.     │
│                                                                      │
│  8. ACID C ≠ CAP C. ACID covers all DB rules;                       │
│     CAP C covers only single-copy consistency.                      │
│                                                                      │
│  9. NoSQL = "Not Only SQL" — 4 main types:                          │
│     Key-Value, Columnar, Document, Graph                            │
│     Each optimized for different data models and access patterns.   │
│                                                                      │
│  DESIGN PRINCIPLE: Choose the right consistency model for each      │
│  part of your system. Not everything needs strong consistency!       │
└──────────────────────────────────────────────────────────────────────┘
```
